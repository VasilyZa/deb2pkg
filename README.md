# deb2pkg — Debian/RPM 包转 Arch Linux PKGBUILD 工具

`deb2pkg` 是一个纯 C++17 命令行工具，可将 `.deb`（Debian/Ubuntu）或 `.rpm`（Fedora/RHEL）安装包自动转换为 Arch Linux 的 **PKGBUILD** 文件，遵循 [AUR 打包规范](https://wiki.archlinux.org/title/PKGBUILD)。

## 功能特性

- **自动类型检测** — 根据文件魔数或扩展名识别 .deb / .rpm 格式
- **纯解压操作** — 不调用 dpkg/rpm 安装命令，不污染宿主操作系统
- **元数据提取** — 从包内部解析软件名、版本号、架构、描述、许可证
- **路径自动修正** — 将 Debian/Fedora 特有的库目录合并到 Arch 标准路径
- **ELF 依赖分析** — 遍历所有二进制程序，解析 .so 动态库依赖
- **智能包名匹配** — 调用 pkgfile 将库文件自动匹配为 Arch 官方包名
- **AUR 规范输出** — 生成的 PKGBUILD 可直接运行 `makepkg` 构建
- **全中文输出** — 控制台所有输出为中文，错误附带解决建议

## 安装

### 依赖

**编译依赖：**
- C++17 编译器（Clang 或 GCC ≥ 7）
- CMake ≥ 3.10
- libarchive 开发库

**运行时依赖：**
- `ar`（binutils 包，系统自带）
- `readelf`（binutils 包，系统自带）
- `pkgfile`（可选，用于自动匹配依赖包名）
- `makepkg`（可选，用于构建安装包）

### 安装所需依赖

```bash
# 安装编译和运行时依赖
sudo pacman -S cmake clang libarchive binutils pkgfile base-devel

# 初始化 pkgfile 数据库（重要！否则无法自动匹配依赖）
sudo pkgfile --update
```

### 编译安装

```bash
# 克隆或下载源码
cd deb2pkg

# 编译
mkdir build && cd build
cmake .. -DCMAKE_CXX_COMPILER=clang++
make -j$(nproc)

# 安装到系统（可选）
sudo make install

# 或直接将可执行文件复制到 PATH 中
sudo cp build/deb2pkg /usr/local/bin/
```

## 基本使用

### 最简单的用法：只需传入包文件

```bash
deb2pkg google-chrome-stable_current_amd64.deb
```

这会在当前目录下创建 `google-chrome-stable-bin/` 目录，其中包含：
- `PKGBUILD` — 生成的打包脚本
- `google-chrome-stable_current_amd64.deb` — 源文件副本
- `data/` — 解压出的程序文件（符号链接指向临时目录）

### 常用选项

```bash
# 指定输出目录
deb2pkg package.deb -o ~/aur-packages

# 自动构建 .pkg.tar.zst 安装包
deb2pkg package.rpm -b

# 详细输出（查看处理过程）
deb2pkg package.deb -v

# 保留临时文件（调试用）
deb2pkg package.deb -v -k

# 完整流程：输出到指定目录 + 自动构建
deb2pkg slack-desktop.rpm -o ~/aur -b
```

### 完整选项列表

| 选项 | 说明 |
|------|------|
| `-o, --output <目录>` | 指定 PKGBUILD 输出目录，默认为当前目录 |
| `-b, --build` | 自动调用 `makepkg -sf` 构建 .pkg.tar.zst |
| `-k, --keep-temp` | 保留临时文件不删除（用于调试） |
| `-v, --verbose` | 输出详细的处理过程 |
| `-h, --help` | 显示帮助信息 |

## 工作流程

当你运行 `deb2pkg package.deb` 时，工具会按以下步骤自动执行：

```
1. 检测包类型
   └→ 通过文件扩展名和魔数判断是 .deb 还是 .rpm

2. 创建临时目录
   └→ 在 /tmp 下创建独立的临时工作目录

3. 解压文件
   ├→ [.deb]  ar x 提取 → 解压 control.tar.* + data.tar.*
   └→ [.rpm]  读取 Header → libarchive 提取 Payload

4. 解析元数据
   ├→ [.deb]  解析 control 文件 (RFC 822 格式)
   └→ [.rpm]  手动解析 RPM Header 二进制结构
   └→ 提取：软件名、版本、架构、描述、许可证

5. 路径修正
   ├→ /usr/lib/x86_64-linux-gnu/* → /usr/lib/*
   ├→ /usr/lib64/* → /usr/lib/*
   ├→ /lib64/* → /usr/lib/*
   └→ 清理空目录

6. ELF 依赖扫描
   └→ readelf -d <ELF文件> → 收集所有 NEEDED soname

7. pkgfile 包名匹配
   └→ pkgfile -s <soname> → 查找所属 Arch 包名 → depends[]

8. 生成 PKGBUILD
   └→ 写入输出目录（格式符合 AUR 规范）

9. [可选] makepkg 构建
   └→ makepkg -sf --noconfirm → .pkg.tar.zst

10. 清理临时文件
    └→ 默认自动删除，--keep-temp 可保留
```

## 生成 PKGBUILD 示例

以下是将 `google-chrome-stable_current_amd64.deb` 转换后生成的 PKGBUILD：

```bash
# Maintainer: Generated by deb2pkg
# 原始包: google-chrome-stable
# 原始格式: Debian
# 生成时间: 2025-06-24 16:30:00

pkgname=google-chrome-stable-bin
pkgver=125.0.6422.60-1
pkgrel=1

pkgdesc='The web browser from Google'
arch=(x86_64)
license=(custom)
url='https://www.google.com/chrome/'
source=('google-chrome-stable_current_amd64.deb')
sha256sums=('abc123def456...')
depends=('glibc' 'gtk3' 'nss' 'libcups' 'libxss')
provides=('google-chrome-stable')
options=(!strip !emptydirs)

package() {
  cd "${srcdir}"
  if [ -d "data" ]; then
    cp -r --no-preserve=ownership data/* "${pkgdir}/"
  else
    echo "错误：未找到 data/ 目录，请先用 deb2pkg 解压该包"
    return 1
  fi
  rm -rf "${pkgdir}/usr/lib/x86_64-linux-gnu" 2>/dev/null || true
  rm -rf "${pkgdir}/usr/lib64" 2>/dev/null || true
  rm -rf "${pkgdir}/lib64" 2>/dev/null || true
}
```

## 常见问题

### Q: 为什么提示 "pkgfile 未安装"？

**A:** pkgfile 是可选但强烈推荐的工具，用于自动将共享库文件名（如 `libc.so.6`）匹配为 Arch Linux 的包名（如 `glibc`）。

```bash
sudo pacman -S pkgfile
sudo pkgfile --update
```

如果不安装 pkgfile，工具会打印所有未匹配的 soname，你需要手动在 PKGBUILD 的 `depends` 数组中添加对应包名。

### Q: 有些库文件没有自动匹配怎么办？

**A:** 终端会列出所有未匹配的 soname：
```
⚠ 警告：以下 3 个共享库未能自动匹配 Arch 包名：
  - libfoo.so.1
  - libbar.so.2
```

你可以：
1. 手动运行 `pkgfile -s libfoo.so.1` 查找包名
2. 在 AUR 搜索：`yay -Ss libfoo` 或访问 https://aur.archlinux.org
3. 如果确实找不到，该库可能已包含在某个基础包中（如 glibc），无需单独添加

### Q: 生成的 PKGBUILD 无法运行 makepkg？

**A:** 检查以下几点：
```bash
# 1. 进入 PKGBUILD 所在目录
cd <pkgname>

# 2. 检查 PKGBUILD 语法
namcap PKGBUILD

# 3. 确认源文件存在
ls -la <source_file>

# 4. 确认 data/ 目录存在
ls -la data/

# 5. 测试构建（不实际打包）
makepkg --noextract --verifysource
```

### Q: 如何手动补全 PKGBUILD？

**A:** 生成的 PKGBUILD 是一个起点，你可能需要手动调整：

1. **license 字段** — 如果显示 `custom`，请根据软件实际许可证修改
2. **depends 数组** — 补充未自动匹配的依赖包名
3. **pkgver** — 版本号可能需要格式调整以符合 Arch 规范
4. **package() 函数** — 如有特殊安装需求，调整文件复制逻辑

### Q: 支持哪些架构的包？

**A:** 支持的架构映射：
| Debian/Ubuntu | Fedora/RHEL | Arch |
|---------------|-------------|------|
| amd64 | x86_64 | x86_64 |
| i386 | i686 | i686 |
| arm64 | aarch64 | aarch64 |
| armhf | armv7hl | armv7h |
| all | noarch | any |

## 路径修正规则

工具会自动修正以下发行版特有的目录路径：

| 原始路径 | 修正后路径 |
|----------|-----------|
| `/usr/lib/x86_64-linux-gnu/` | `/usr/lib/` |
| `/usr/lib/i386-linux-gnu/` | `/usr/lib/` |
| `/usr/lib/aarch64-linux-gnu/` | `/usr/lib/` |
| `/usr/lib64/` | `/usr/lib/` |
| `/lib64/` | `/usr/lib/` |
| `/lib/x86_64-linux-gnu/` | `/usr/lib/` |

注：修正是通过文件移动（rename）实现的，不产生额外磁盘占用。

## 技术架构

```
src/
├── main.cpp                    # 入口：CLI 参数解析 + 主流程编排
├── core/                       # 基础层
│   ├── types.h                 # 核心数据结构
│   ├── errors.h                # 错误码 + 中文解决建议
│   ├── logger.h/cpp            # 中文日志工具
│   └── executor.h/cpp          # 进程调用封装（fork+exec）
├── extract/                    # 文件解压模块
│   ├── archive_reader.h/cpp    # libarchive 封装
│   ├── deb_extractor.h/cpp     # .deb 解压
│   └── rpm_extractor.h/cpp     # .rpm 解压
├── metadata/                   # 元数据读取模块
│   ├── deb_control_parser.h/cpp # deb control 文件解析
│   └── rpm_header_parser.h/cpp  # RPM Header 二进制解析
├── transform/                  # 转换模块
│   ├── path_fixer.h/cpp        # 路径修正
│   └── elf_dependency_scanner.h/cpp # ELF 依赖分析
├── pkgfile/
│   └── pkgfile_resolver.h/cpp  # pkgfile 包名匹配
└── generate/                   # 生成模块
    ├── pkgbuild_writer.h/cpp   # PKGBUILD 生成
    └── makepkg_builder.h/cpp   # makepkg 构建
```
